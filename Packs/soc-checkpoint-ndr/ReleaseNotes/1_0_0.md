## [1.0.0] - May 2026

### Added

#### SOC CheckPoint NDR - Behavioral Alerts — New correlation rule

Replaces the OEM-default Check Point NDR correlation rule, which produced alerts with empty `alert_fields`, missing source/data tags, and broken severity mapping. Symptoms: SOC Framework playbooks hung in `pending` state on NDR alerts because Foundation Product Classification had no `DS:` tag to match.

**What the rule produces:**

- Source / destination IPs extracted from `_alert_data.sourceips[0]` and `_alert_data.destinationips[0]`, mapped to canonical `action_remote_ip` / `action_local_ip` columns. For inbound IPS-style alerts (the only observed pattern for Behavioral), source = external attacker, destination = internal target.
- Destination port from `_alert_data.dstports[0]`.
- Severity mapped from Check Point's `SEV_0X0_X` ladder to XSIAM standard (`Informational/Low/Medium/High/Critical`). Only `SEV_020_LOW` and `SEV_030_MEDIUM` observed today; full ladder included for resilience.
- Alert category derived from the first dot-segment of the `insight` field (`Behavioral.Geo.Mexico` → `Behavioral`). Subcategory preserved separately.
- Bytes-sent / bytes-received from `_alert_data`, surfaced as alert fields.
- Check Point's `top_protection_name` (when present) appended to the alert name as a qualifier.
- `originalalertconfidence` populated from Check Point's `probability` field (note: this is a vendor-relative score, not the XSIAM 0–100 scale).
- `originalalertfilter` preserves Check Point's filter expression for forensics.
- `externallink` populated from `_alert_data.externallink` for one-click pivot to the Check Point console.

**Tags injected:**

- `DS:Check Point/NDR` — Foundation Product Classification reads this and normalizes to `ds_check_point_ndr`, keying into `SOCProductCategoryMap_V3` for Network-category routing.
- `DOM:Security` — domain tag (the source `alert_domain` of `DOMAIN_HEALTH` is intentionally coerced to `DOMAIN_SECURITY` for consistent Workflow dispatch; revisit once a Health-domain Workflow exists).

**Suppression:** 1-hour window keyed on `originalalertid` to deduplicate the same NDR alert re-firing on subsequent polling cycles.

**MITRE:** Empty for Behavioral alerts (Check Point NDR doesn't carry MITRE on this insight type). The four required `mitretactic*` / `mitretechnique*` `alert_fields` entries are present and resolve to null per the passthrough subtype contract.

### Notes

- Foundation Product Classification depends on `tags` being populated. If the `alert_fields → tags` mapping doesn't propagate to `issue.tags` on the resulting alert in some environments, the fallback is to add a small modeling rule that sets `xdm.alert.tags` at ingestion time. Validate on tenant before assuming routing works end-to-end.
- The Check Point NDR alert description sometimes references IPs that don't match the `sourceips` / `destinationips` fields — the description describes the originating behavioral pattern while the alert is a narrowed slice. Always trust the structured IPs for response actions; the description is analyst context only.
